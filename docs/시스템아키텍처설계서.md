## 2. 프론트엔드 아키텍처

### 2.1 설계 원칙

- **컴포넌트**: 독립적·재사용 가능한 단위
- **로직**: Composition API + **Composable**로 상태·비즈니스 로직 캡슐화
- **타입**: TypeScript로 시그니처·도메인 모델 명확화
- **구조**: 도메인/기능 중심 + 책임 분리

### 2.2 폴더 구조

```
frontend/src/
├── assets/           # 정적 자산
├── components/       # Vue 컴포넌트
│   ├── ui/           # 공통 UI (BaseCard, BaseModal 등)
│   ├── home/         # 홈 도메인
│   ├── calendar/     # 일정 도메인
│   ├── courses/      # 강좌 도메인
│   ├── profile/      # 프로필 도메인
│   └── gym/          # Gym 도메인
├── composables/      # 로직 캡슐화
│   ├── calendar/     # useCalendarPeriod, useEventDetails
│   ├── gym/          # useGymManagement
│   └── course/       # useCourseList
├── layouts/          # AdminLayout 등
├── router/           # routes.ts
├── services/         # API·비즈니스 로직
│   ├── core/         # access, audit, utils, userUtils
│   └── domain/       # schedule, gym, profile, class, user
├── stores/           # Pinia 스토어
├── types/            # TypeScript 타입 (도메인별 분리)
├── views/            # 라우트별 페이지
└── i18n/             # ko.ts, en.ts
```

### 2.3 데이터 흐름

```
[View] ←→ [Composable / Store] ←→ [Service] ←→ [Firebase]
   │              │                    │
   │              │                    └── Firestore, Auth, Storage
   │              └── Pinia (auth, schedule, class, trainee, profile, theme, ui)
   └── UI 바인딩, 이벤트 핸들링
```

### 2.4 서비스 레이어

| 서비스 | 역할 |
|--------|------|
| **core/access.ts** | AccessActor, assertCanAccessUserData |
| **core/audit.ts** | AuditPayload, writeAuditLog |
| **core/utils.ts** | chunkByTen 등 유틸 |
| **core/userUtils.ts** | AppUser, mapUsers, 역할 유틸 |
| **domain/scheduleService.ts** | 일정 CRUD, 세션 완료, 클래스 로그 |
| **domain/gymService.ts** | Gym CRUD, 트레이니 조회 |
| **domain/profileService.ts** | 트레이너/바디 프로필 |
| **domain/classService.ts** | 클래스 생성, 트레이니 추가/제거 |
| **domain/userService.ts** | 트레이니/트레이너/매니저/스태프 CRUD |
| **courseService.ts** | 강좌 CRUD, 신청/승인 |
| **toolService.ts** | 기구 사용법 CRUD |

### 2.5 프론트엔드 설계 원칙 및 개선 방향

#### 2.5.1 현재 구조 요약

| 영역 | 현재 상태 | 비고 |
|------|-----------|------|
| **views/** | 페이지 단위 뷰. 일부는 가벼움(DashboardView), 일부는 무거움(CalendarView, GymManagement, CourseListView) | 무거운 뷰에 로직·상태 집중 |
| **components/** | home / courses / calendar / profile / gym / ui 로 구분 | 기능 도메인 + UI 공통 컴포넌트 혼재 |
| **stores/** | auth, schedule, class, trainee, profile, theme, ui | 도메인별 스토어 분리 |
| **composables/** | useCalendarPeriod, useGymManagement, useCourseList, useEventDetails 등 | 도메인별 로직 캡슐화 완료 |
| **types/** | schedule, user, gym, trainer, profile, tool, course 등 도메인별 분리 | types/index.ts에서 re-export |

#### 2.5.2 개선 방향 (적용 완료)

- **서비스 레이어**: 도메인별 서비스 분리 완료 (core/, domain/)
- **Composables**: 캘린더·Gym·강좌·일정 상세 로직 composable로 이전
- **뷰/컴포넌트**: "표시 + composable 호출" 수준으로 책임 정리
- **타입**: 도메인별 types/*.ts 분리, 서비스에 명시적 반환 타입 적용

#### 2.5.3 적용 완료 상세

- **core/**: access.ts, audit.ts, utils.ts, userUtils.ts
- **domain/**: scheduleService, gymService, profileService, classService, userService
- **composables/**: calendar/, gym/, course/ 하위 useCalendarPeriod, useGymManagement, useCourseList, useEventDetails

## 3. Firestore 데이터 모델

### 3.1 컬렉션 구조

| 컬렉션 | 용도 | 주요 필드 |
|--------|------|-----------|
| **users** | 사용자 | email, role, lvl, gymId, sessionCount, expiryDate, profileComplete, fcmToken |
| **gyms** | 지점 | name, location, phone, managerEmails |
| **classes** | 그룹 클래스 | trainerId, traineeIds, name |
| **schedules** | 일정 | type, userEmail, status, start, end, signature, workoutLogRef |
| **workoutLogs** | 운동 상세 | scheduleId, exercises (종목, 세트, 무게, 횟수) |
| **bodyProfiles** | 바디 프로필 | userId, weight, bodyFat, date |
| **ticketHistory** | 수강권 이력 | userId, action, count, reason |
| **trainerProfiles** | 트레이너 프로필 | userId, intro, expertise, career |
| **profileHistory** | 프로필 변경 이력 | - |
| **adminAuditLogs** | 관리자 감사 로그 | actor, action, target |
| **toolUsage** | 기구 사용법 | title, mediaItems, visibility |
| **courses** | 강좌 | trainerId, gymId, schedule, attendees |
| **courseApplications** | 강좌 신청 | courseId, traineeEmail, status |

### 3.2 Firestore 보안 규칙 요약

| 컬렉션 | 읽기 | 쓰기 |
|--------|------|------|
| users | 로그인 | 본인/SITE_ADMIN/MANAGER |
| adminAuditLogs | SITE_ADMIN | SITE_ADMIN/MANAGER (create만) |
| ticketHistory | 로그인 | TRAINER/SITE_ADMIN |
| classes | 로그인 | TRAINER/MANAGER/SITE_ADMIN |
| schedules | 로그인 | 트레이너/본인(개인)/SITE_ADMIN |
| workoutLogs | 로그인 | TRAINER/SITE_ADMIN |
| gyms | 로그인 | MANAGER/SITE_ADMIN |
| courses | 로그인 | TRAINER/SITE_ADMIN |
| courseApplications | 로그인 | 본인(create)/트레이너(delete) |
| 기타 | 로그인 | false (명시 규칙 필요) |

## 5. 배포 아키텍처

### 5.1 현재 배포

| 대상 | 플랫폼 | 설정 |
|------|--------|------|
| Frontend | Vercel | `cd frontend && npm run build`, 출력 `frontend/dist` |
| SPA 라우팅 | Vercel | rewrite → index.html |
| Firestore 규칙 | Firebase | firestore.rules |
| Cloud Functions | Firebase | functions/ |

### 5.2 모바일 배포 (Capacitor)

- Vue 웹 앱 → Capacitor → iOS/Android 네이티브 프로젝트
- FCM, GoogleService-Info.plist, google-services.json 설정 필요
- 상세: `Deployment_Mobile.md` 참고

## 6. 보안 아키텍처

### 6.1 인증

- Firebase Authentication (이메일/비밀번호)
- JWT 기반 세션, 클라이언트에서 uid·email 활용

### 6.2 권한 제어

- **클라이언트**: 라우트 meta (requiresAuth, requiresManager, requiresSiteAdmin, requiresTrainer)
- **서버**: Firestore Security Rules (isSignedIn, isSiteAdmin, isManager, isTrainer, isOwner)
- **서비스**: assertCanAccessUserData (AccessActor)

### 6.3 감사

- adminAuditLogs: 관리자·매니저 작업 이력 기록
